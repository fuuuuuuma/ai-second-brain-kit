## Vault の構造

- `Memory.md` … **【必須】** オーナーの引き継ぎ書（誰で・何をしていて・どんな判断基準か）。**事実の置き場**
- `Home.md` … **【必須】** Vaultの玄関（ハブ／目次）。主要ノートへのリンク集
- `raw/` … オーナーが素材を放り込む場所。**読み取り専用（削除・改名しない）**
- `wiki/` … あなたが構造化して書く知識ベース。`wiki/index.md` は全ページの目次
- `wiki/moc/` … Map of Content（テーマ別の地図）
- `reports/` … 問いへの答え・調べものの成果物
- `daily/` … デイリーノート
- `outputs/` … ブログ・台本など最終成果物（**ここだけフォルダで管理してよい**）
- `rules/` … 運用ルール（下記の必読ファイル）
- `templates/` … ノート雛形

## 起動時の必読手順（毎回）

新しいセッションを始めたら、回答の前に必ずこの順で読む：

1. この `CLAUDE.md`（ルール＝憲法）
2. `Memory.md` … **【必須】** オーナーは誰で、何をしていて、どんな判断基準か（事実・前提）
3. `rules/corrections.md` … オーナーから受けた修正指示の蓄積（**ここに書かれた指示は恒久的に守る**）
4. `rules/mistakes.md` … 過去のやらかしと再発防止ルール
5. `wiki/index.md` … 今ある知識の目次（全体像は `Home.md` からも辿れる）

## 絶対に守る運用ルール

- **修正の資産化**：オーナーに訂正・要望を伝えられたら、その場で `rules/corrections.md` に3行（日付 / 指摘内容 / 今後どう動くか）で追記する。同じことを二度言わせない。
- **失敗の記録**：同じ失敗を指摘されたら `rules/mistakes.md` に追記し、再発防止ルールを書く。
- **素材の構造化**：`raw/` に素材が来たら読み、概念ごとに `wiki/` のページへ構造化し、関連ページへ `[[リンク]]` を張り、`wiki/index.md` を更新する。
- **答えはファイルに**：重要な問いへの答えは `reports/` にファイルで残し、根拠にした `wiki/` ページへリンクする。チャットに流して消さない。
- **Memory と Home を最新に保つ**：オーナーの前提・進行中・判断基準が変わったら `Memory.md` を更新する。重要ノート・MOC・進行中の案件が増えたら `Home.md` にリンクを1行足す。
- **整理しすぎない**：フォルダで細かく分類しない（`outputs/` を除く）。分類より `[[リンク]]` と命名で繋ぐ。命名は `rules/naming.md` に従う。
- **Obsidian 構文で書く**：ノートは Obsidian Flavored Markdown で書く。内部リンクは `[[wikilink]]`、メタは frontmatter（type/status/date/topic/tags）、強調は callout（`> [!note]`）。Webの取り込みは Defuddle でクリーンに。
- **週次の手入れ**：週に一度 `rules/lint.md` の観点で `wiki/` を点検し、矛盾・重複・欠落・古い情報を洗い出して報告する。
- **破壊的操作の前に確認**：削除・上書き・大量改名・外部送信の前は、必ずオーナーに確認する。
- **事実は捏造しない**：日付・数値・固有名詞・引用は、確証がなければ「未確認」と明示する。`raw/` に出典がない断定をしない。
